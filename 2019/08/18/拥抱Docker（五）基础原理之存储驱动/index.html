拥抱Docker（五）基础原理之存储驱动

发表于 2019-08-18 | 分类于 Docker |

概述

  在使用Dockerfile制作一个Docker镜像的时候，通常会使用FROM引入一个基础镜像，在拥抱Docker（二）Dockerfile制作镜像中也说过，”Dockerfile的每一条指令都是对镜像做出一次修改，每次修改都会提交一个新的镜像层“。为了了解镜像究竟是如何分层的，我们需要进一部了解Docker内部的文件存储系统。

AUFS

  Docker最开始采用AUFS作为文件存储驱动，AUFS全称Advanced Multi-Layered Unification Filesystem。在介绍AUFS之前，先说说它的前身UnionFS，一种联合文件挂载系统，通俗地讲就是把多个不同物理位置的目录合并挂载到同一个目录上。UnionFS用到了一个重要的资源管理技术，写时复制（copy on write简称CoW），这种技术能够对可修改资源实现高效的复制，如果一个资源是重复的，且在没有任何修改的情况下，并不需要立即创建一个新的资源，而是共享旧资源的实例。AUFS继承UnionFS的思想，并完全重写了UnionFS，在提高了可靠性和性能的同时，也引入了一些新的概念，比如可写分支平衡。

Btrfs

  Btrfs号称下一代写时复制文件系统，目标是取代Linux目前的ext3文件系统，目前已经并入Linux内核中，Btrfs具有以下特质（只列举部分）：

  • 扩展性(scalability) Btrfs将磁盘空间划分为多个chunk，每个chunk可以使用不同的磁盘空间分配策略，这样做可以灵活地分配设备空间，用户在系统中增加新的磁盘之后，可以使用Btrfs的命令将该设备添加到文件系统中。
  • 数据一致性 (data integrity) Btrfs采用CoW事务技术来保证文件系统的一致性，并且支持checksum，能够快速检测文件是否受损，如果损坏还能尝试修复，这样极大的提高数据的准确性，避免因硬件故障导致Silent corruption的出现。
  • 支持创建快照 (snapshot）和克隆 (clone），可以方便地管理多个物理设备。

Device mapper

  Device mapper是Linux2.6内核中提供的一种从逻辑设备到物理设备的映射机制，在该机制下，用户可以很方便的根据自己的需要制定实现存储资源的管理策略，如条带化，镜像，快照等，其中Thin-Provisioning Snapshot，这是Docker使用DeviceMapper中最重要的模块。Thin-Provisioning（储存资源随需分配）是一项利用虚拟化方法减少物理存储部署的技术，可最大限度提升存储空间利用率，举个例子来说，比如你购买了1TB的云空间容量，然后只存了100GB的内容，实际上分配给你的空间就是100G，并不是实际上的1TB。Thin-provisioning Snapshot 结合Thin-Provisioning和Snapshot两种技术，允许多个虚拟设备同时挂载到一个数据卷以达到数据共享的目的，结合CoW技术，在写操作的时候触发实际的存储分配。

OverlayFS

  OverlayFS是一种流行的Union FS，类似于AUFS，不过OverlayFS是内核提供的文件系统，和AUFS的多层不同，Overlay只有两层：一个upper文件系统和一个lower文件系统，分别代表Docker的镜像层和容器层。当需要修改一个文件时，使用CoW将文件从只读的lower复制到可写的upper进行修改，结果也保存在upper层，并通过merged的目录公开（挂载点）。在Docker中，底下的只读层就是image，可写层就是Container。结构如下图所示：

Docker为OverlayFS提供两种存储驱动，Overlay和Overlay2。Overlay2是Overlay的更新版本，在4.0以上内核添加了Multiple lower layers in overlayfs的特性，可以支持多个lower layers挂载，现在centos（我的是centos 7.6.1810）中默认使用的就是Overlay2，后面的Docker文件示例中也会以此举例。

ZFS

  ZFS也是号称下一代文件存储系统，它从根本上改变了文件系统的管理方式。ZFS 完全抛弃了“卷管理”，不再创建虚拟的卷，而是把所有设备集中到一个称之为zpools的存储池中来进行管理。在Docker中，首先从zpool分配一个ZFS文件系统给镜像的基础层，而其他镜像层则是这个ZFS文件系统快照的克隆。快照是只读的，而克隆是可写的，用于存储与上一层的差异。当容器启动时在镜像的最顶层生成一个可写层，容器的读写层都是通过这个顶层的快照创建克隆层而来。如下图所示：

VFS

  VFS不是一个联合文件存储系统，每一层都是一个真实的目录，也没有CoW的支持，要创建一个新图层，先前的图层会进行“深层复制”。与其他存储驱动程序相比，这会导致磁盘性能下降和磁盘使用空间更多。但是，它强大、稳定，适用于各种环境，可以用作在测试环境中验证其他存储后端的机制。

Docker文件存储示例

  上面几乎介绍了Docker所有支持的文件存储驱动，那么Docker真实的镜像、容器的文件存储目录结构是怎样的呢？上面说Docker的镜像是一个分层的结构，其实每一个Docker镜像都是由一个一个可读（read-only）的layer组成的。就像我们上面演示的google.txt示例，修改mount中的google.txt不会影响google文件夹中的google.txt，镜像的每一层layer有了这种能力，那么镜像每一层不仅可以重用，而且修改某个镜像层可以保证不会影响其他镜像。
