从mybatis到mybatis-spring-boot-starter

发表于 2020-12-22 | 分类于 mybatis |

MyBatis

  MyBatis 是一款优秀的持久层框架，可以通过简单的 XML 或注解来配置映射接口进行一些数据库操作。与同流行的持久层开发框架Hibernate相比，个人觉得MyBatis的优点如下：简单（容易掌握）、高效（手写SQL，对复杂查询更友好）。最近因为公司项目，花了几天稍微看了一下MyBatis的源码，以前虽然对MyBatis已经有了不少接触和了解，但是从没有尝试完整从源码的角度去解构。下面从一段MyBatis的入门代码开始，借以分析一些核心部件：

Configuration

以上是采用注解的方式来使用Mybatis的例子，从Configuration开始讲起，Configuration包含了MyBatis所有的配置信息，以及映射的Mapper接口，比如上面BlogMapper会被加入到Configuration中。如果是在xml中配置的Mapper，会通过SqlSessionFactoryBuilder加载到Configuration中。

MappedStatement

解析xml或者注解中的SQL语句，其实就是组装成MappedStatement，简单来说MappedStatement就是存放SQL的地方。

SqlSession

SqlSession是一次访问数据库的会话，内部封装了一套数据库增删改查的方法。SqlSession是MyBatis的Api，正确理解SqlSession才能真正用好MyBatis。按照官方文档来说，每个线程都应该有它自己的 SqlSession 实例，因为 SqlSession 不是线程安全的。注意，绝对不能将 SqlSession 实例的引用放在一个类的静态域，甚至一个类的实例变量也不行。官方推荐在Web应用中，每次收到 HTTP 请求，就可以打开一个 SqlSession，返回一个响应后，就关闭它。SqlSession是一个接口，其主要的实现为DefaultSqlSession。

Executor

那么继续从SqlSession#selectList开始：

从Configuration获取到了MappedStatement（相当于拿到SQL了），使用Executor执行。Executor是一个接口，负责数据库操作调度、缓存，主要有5个实现类，继承关系如下：

  1. BaseExecutor 抽象基类，模板模式封装一些Executor共有方法。
  2. CachingExecutor 封装缓存，参见TransactionalCacheManager实现
  3. SimpleExecutor 默认使用SimpleExecutor，也是最简单直接的执行器。
  4. BatchExecutor 批量操作执行器，在commit才会提交数据，提高性能
  5. ReuseExecutor 可重用执行器，和SimpleExecutor相比主要是多了个Statement缓存

StatementHandler

StatementHandler是一个接口，主要作用是调用Statement执行SQL语句，它主要有5个实现，继承关系如下：

  1. BaseStatementHandler 和Executor结构类似，BaseStatementHandler是抽象类，封装了一些公用、模板方法。
  2. CallableStatementHandler 调用存储过程
  3. PreparedStatementHandler SQL语句预编译后执行
  4. SimpleStatementHandler SQL语句无需预编译，直接执行
  5. RoutingStatementHandler 代理、路由其他StatementHandler

Cache

MyBatis的缓存分为一级缓存、二级缓存，逻辑如下：

  1. 一级缓存在BaseExecutor已经提过，一级缓存的逻辑非常简单，会在update或者commit（LocalCacheScope=SESSION）的时候清空。一级缓存的实现PerpetualCache，实际上是在内部维护了一个HashMap。
  2. 二级缓存在CachingExecutor，默认是开启的（但是需要配置缓存实现，比如可以在Mapper上加@CacheNamespace注解即可使缓存生效）。一级缓存的特点是只能在SqlSession内部，而二级缓存可以应用到多个SqlSession之间，注意二级缓存只有在SqlSession commit或者close之后才会缓存。二级缓存的实现非常多，比如：LruCache、FifoCache、BlockingCache……不再一一举例。
ParameterHandler

ParameterHandler负责给Statement设置参数，在SimpleExecutor#doQuery中，在调用StatementHandler#query之前，会先调用StatementHandler#prepareStatement，这里做了两件事：初始化Statement和给Statement设置参数。这一步在StatementHandler#parameterize执行（SimpleStatementHandler#parameterize是个空方法），这个方法实际上调用ParameterHandler#setParameters。ParameterHandler是一个接口，其实现类也只有一个即DefaultParameterHandler，一个简单的设置参数的动作，为什么MyBatis还要搞出一个ParameterHandler？这一切其实和下面要说的Inteceptor有点关系。

Inteceptor

Inteceptor即MyBatis的插件，通过Inteceptor可以做很多的事情，比如可以动态修改SQL、监测SQL的执行耗时统计、权限拦截等等。MyBatis的Plugin依靠Inteceptor完成，支持对Executor、StatementHandler、PameterHandler和ResultSetHandler接口进行拦截。上面对于ParameterHandler的构造，很大一部分也是为了满足插件系统对于参数设置这一步的介入。Inteceptor在这几个组件创建的时候其实就埋伏好了，贴一段Configuration的代码：

Transaction

MyBatis的Transaction实现非常简单，有一个Transaction接口封装了所有的事务操作如commit、rollback等，分别有两个实现：JdbcTransaction和ManagedTransaction：

  • JdbcTransaction 简单地调用JDBC的方法操作事务
  • ManagedTransaction 将事务的整个生命周期交给其他容器来管理，本身不做什么，所以commit、rollback都是空的方法。
