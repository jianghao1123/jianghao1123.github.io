    DDD实践笔记

    Posted on 2021-01-17 In DDD

      在使用领域驱动设计（DDD）做了几个项目之后（实际上更多的是借鉴DDD的战术设计），感觉自己更加深入地了解了DDD以及DDD相关架构模式（特别是六边形架构）的一些思想，再结合项目之后的归纳和总结，不论优点和缺点，总归有了一些收获，于是决定写一个笔记，将整个实践过程中采纳的方案、要点、还有一些弯弯绕绕的东西都记录下来。

    架构设计

      架构设计主要是探讨四层架构模式以及之前项目的一些模块设计，如果对DDD的一些概念或者术语还不是很清楚，可以参考领域驱动设计（DDD）笔记。在做架构设计的时候，不管最后使用了何种方式，我们只需要记住几个小目标，具体参照整洁架构（The Clean Architecture）：

    • 框架分离 业务不受框架的限制
    • 可测试性 业务逻辑独立测试，不依赖于UI、数据库、Web服务等其他外部条件
    • 独立于UI UI层的替换和修改不会修改其他系统模块
    • 独立于DB 数据库存储的独立、可替换性
    • 与外部结构分离 系统业务逻辑不需要知道任何外部结构
    分层架构

      分层架构是所有架构的始祖，DDD分层架构经过一些优化和解耦后形成了如下的四层架构：
    在传统的四层架构中，基础层是被其他层依赖的，这种依赖方式按照分层架构的思想来说，基础设施层才应该是核心，但实际上不是，所以传统的四层架构是有问题的。后来传统的四层架构又通过依赖倒置的方式转变为以领域层为核心的四层架构，我们可以通过在领域层中定义接口，然后在基础设施层实现的方式来解耦基础层。

    模块设计

      按照DDD四层架构的思想，我们的项目分为以下四个模块：

    • infrastructure 基础设施层
    • interface 接口层
    • application 应用层
    • domain 领域层

    很多DDD项目也是基于此进行模块或者包的设计，比如领域层的实现放在 org.matrxi.ddd.demo.domain包下，这种是以包的方式进行模块管理，适合比较小的项目，当然为了更加细致地拆分不同模块，我们通常使用Module化的方式拆分，比如单独会有一个domain的模块来存放domain相关的业务。不过需要注意，因为领域层可能不单单只有一种领域（核心域），可能包含了核心域、子域、支撑域等等（从项目团队维护的角度考虑，初期不会拆分的这么细致）；或者从 interface 接口层考虑，也不单单只有Web接口，可能会以微服务化的方式提供RPC服务，或者以消息中间件的方式来消费数据。所以单一的模块化管理也可能会出现因为模块数量迅速膨胀而导致整个项目模块混乱的情况。基于此考虑，我觉得DDD的四层不应该作为直接模块来使用，而应该作为模块的分类，如下：
    ![](DDD实践笔记/project_ architecture.jpg)

      使用四层架构作为顶层模块分类，具体的模块放在各自层的文件夹中，比如domain下可以有多个领域模块，infrastructure下也会有多种数据库仓储和缓存的实现，这样整个项目结构看起来会更加的清晰。

    战略设计

      因为DDD的战略设计和业务之间强关联，每一种业务各自的侧重点大相径庭，所以很难进行详尽的总结和表述，我没法记录太多关于战略设计的东西，我觉得主要还是要对业务有全局的视野和整体把控，这样才能理解领域中核心域、子域、支撑域、通用子域的一些概念，区分轻重缓急。

    案例分析

      规划好了架构和项目模块，我们来举个例子，以账户转账功能作为整个项目的需求点，假设我们拆解出如下步骤：

    1. A账户转钱给B账户，先从数据库中查找到对应的A/B账户；
    2. 扣款校验，确保A账户有钱；
    3. A账户出账，B账户进账，保存到数据库；
    4. 发送转账成功事件，用于对账审计；

    以上只是虚拟的转账业务功能，不要较真。

    domain

      首先在domain/core模块下新建Account:

    Account是一个很简单的账户实体，其中又包含了余额值对象，以及一些账户的领域行为。这里不论Account建模是否合适，在项目初期我们就应该形成DDD领域建模的理念，不要老是将关注的重点放在数据库层，不要急急忙忙地去设计数据库的列和关联，然后再将数据库的列回写到领域对象中。这会导致领域层实体（Entity）只是一个包含大量getter和setter方法而缺乏实际行为的贫血对象，这样的做法不是DDD的做法，DDD应该优先规划好实体、值对象，使用更加面向对象的方式去建模。

    领域建模第一步，将开发的关注重点转移到业务领域建模上，而不是数据实现。

      有了Account实体是不够的，因为一个转账功能需要两个账户，所以需要借助领域服务来实现，我们又要在domain/core模块新建TransferService：

    整个TransferService的功能设计非常简单，只是扣款和进款，至于校验账户有效性的功能我觉得应该交给账户本身，比如账户在withdraw的时候本身就应该考虑是否有足够的余额进行出账，如果没有可以通过异常的方式进行中断处理。

    问题

    文档

      通常使用swagger作为文档，因为需要对DTO层的一些字段提供说明，所以需要在application/api层引入swagger依赖，这里也是稍有瑕疵的地方，不过只需要依赖swagger-annotations即可。

    DTO归属

      关于DTO应该放在application还是interface是我比较纠结的问题，目前我的做法是放到了application层，但是也导致了swagger依赖侵入application层的一些问题，显得并不那么优雅。在DDD - which layer DTO should be implemented中高票回答推荐将DTO放在Domain Service Layer（应该相对于application层），不过在DDDSample中DTO归属于用户接口层，只是目前未尝试过。
