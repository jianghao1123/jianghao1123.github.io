RocketMQ之旅（一）概述

发表于 2018-08-18 | 分类于 RocketMQ之旅

  本文基于rocketmq-all-4.3.0。RocketMQ是阿里巴巴开源的分布式消息中间件，使用java语言编写。下面是整个项目模块：
  简单介绍下项目各个模块的用途：

  1. broker 是整个消息服务的中心，负责收发消息，连接consumer和producer。
  2. client 集结了consumer和producer
  3. common 提供通用基础方法给其他模块调用
  4. dev 是一个版本控制脚本不用管
  5. distribution 提供了一些部署的脚本
  6. filter 过滤器，提供SQL92标准的过滤方式
  7. logappender 日志模块
  8. logging 也是日志
  9. namesrv 注册中心，管理broker和topic信息
  10. openmessaging openmessaging是由阿里巴巴发起的分布式消息领域的国际标准，这个包主要提供openmessaging标准的部分实现
  11. remoting 是一个通用的网络通信基础模块，基于netty实现
  12. srvutil 工具包，里面只有三个类
  13. store 消息相关存储服务
  14. style 统一代码风格不用管
  15. test 测试相关
  16. tools 工具类，提供一些后台管理的工具

  模块众多，重点还是关注broker、client、namesrv、remoting、store这几个模块，这些也是一个消息中间件的核心模块。

发表于 2018-08-08 | 分类于 Netty之旅

PoolChunk分配小内存过程

  1. 在Netty中内存以页为最小单位划分，假设分配2048KB大小的内存，如果使用PoolChunk直接分配一页，那真是太浪费了。参照Slab机制，Netty将一页内存进行更细致化的管理，使用PoolSubpage来分配小于一页的内存。
  2. PoolChunk内部维护一个PoolSubpage的数组subpages，数组大小为1 << maxOrder即默认为2048，相当于PoolChunk二叉树的叶子节点数量。
  3. PoolSubpage每次分配内存的时候，只会去二叉树的叶子节点搜寻，当某一个节点被选中时，会根据节点的id，找到subpages数组种对应的PoolSubpage实例，如果该实例为空，则新建一个，然后使用PoolSubpage进行内存分配。看一下PoolChunk的内存分配入口allocateSubpage方法：

Netty之旅（二）ByteBuf引用计数

发表于 2018-07-28 | 分类于 Netty之旅

引用计数

  当我们创建一个对象的时候，对象的引用计数为1。当有一个其他对象指向该对象时，通过retain该对象使其引用计数+1，如果某个对象不再指向该对象时，通过release该对象使其引用计数-1。当该对象的引用计数为0的时候，说明该对象可以被销毁了。
  Java既然已经有了很优秀的内存回收算法了，为什么Netty还要使用引用计数？究其原因，可以从以下两点分析：

  1. ByteBuf可以从直接内存上分配，直接内存不会由JVM直接控制。创建和释放Direct Buffer的代价要比Heap Buffer高很多，使用引用计数可以大大提高内存的释放效率。
  2. 池化Bytebuf也是提高Netty内存分配的一种手段，为了更方便的管理这个对象池，使用引用计数也是一个好的办法。

Netty之旅（一）ByteBuf简介

发表于 2018-07-28 | 分类于 Netty之旅

NIO Buffer缺陷

  1. Buffer设计精简，只有一个position。读写切换的时候需要使用者手动去调用flip()和rewind()，其position和limit的含义，取决于当前Buffer是处于读还是写模式，无疑为整个Buffer编程增加难度。
  2. ByteBuffer长度固定，一旦长度超出limit就会抛出IndexOutOfBoundsException，这就意味着在为ByteBuffer分配容量的时候，你必须精确地知道数据的长度，或者明确知道不会超出某个长度。当然，你也可以自己为ByteBuffer封装功能，实现类似数组扩容的逻辑。
